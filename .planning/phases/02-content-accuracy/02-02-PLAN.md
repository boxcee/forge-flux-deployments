**Introduction paragraph:** Brief note that the app returns standard HTTP status codes. Check `forge logs` for detailed error messages.

**Section: Webhook returns 401 (Authentication Failed)**

Two subsections:

*FluxCD (HMAC):*
- Symptom: All FluxCD webhooks return 401
- Checklist:
  1. Verify `WEBHOOK_SECRET` is set: `forge variables list --environment production`
  2. Verify the secret matches the Kubernetes secret token value exactly
  3. Verify the Provider uses `type: generic-hmac` (not `generic`)
  4. Check that the `X-Signature` header is present (Flux sends `sha256=<hex>` format)
- Debug: `forge logs --environment production` -- look for "HMAC verification failed"

*ArgoCD (Bearer Token):*
- Symptom: All ArgoCD webhooks return 401
- Checklist:
  1. Verify `ARGOCD_WEBHOOK_TOKEN` is set: `forge variables list --environment production`
  2. Verify the token matches the `Authorization: Bearer <token>` header value in `argocd-notifications-cm`
  3. Check header format is `Bearer <token>` (with space, no quotes around token)
- Debug: `forge logs --environment production` -- look for "Bearer token verification failed"

**Section: Deployments not appearing in Jira (silent 204)**

This is the most confusing failure mode -- no error, just nothing happens.

Three causes:
1. **Missing `jira` annotation:** The app skips events without a `jira` annotation (returns 204). Check that the HelmRelease/Application has the annotation set. For FluxCD, use either `event.toolkit.fluxcd.io/jira` or the short key `jira`. For ArgoCD, use `jira`.
2. **Ignored event reasons (FluxCD only):** `UninstallSucceeded` and `DependencyNotReady` events are silently skipped (204). This is intentional -- uninstalls are not deployment state transitions, and `DependencyNotReady` is transient noise.
3. **No matching Alert eventSource (FluxCD):** If the Alert `eventSources` filter does not match the resource kind/name, Flux never sends the webhook.

- Debug: `forge logs` -- look for "No jira annotation -- skipping" or "Ignored reason".

**Section: Webhook returns 400 (Bad Request)**

Two causes:
1. **Missing `env` annotation:** The app requires an `env` annotation. Returns 400 with body "Missing env annotation". Add the annotation to your HelmRelease or Application.
2. **Malformed JSON:** The request body is not valid JSON. Rare with standard CD tool configurations. Check that the ArgoCD notification template body is valid JSON.

**Section: Webhook returns 502 (Upstream Error)**

- Symptom: Webhook returns 502
- Causes: Jira Cloud API rejected the deployment payload
- Checklist:
  1. Verify the app has the `write:deployment-info:jira` scope (should be granted at install)
  2. Check if Jira Cloud is experiencing an outage (status.atlassian.com)
  3. Verify the Jira site is accessible
- Debug: `forge logs` -- look for "Jira API call failed" with the upstream error message

**Section: Common mistakes quick reference**

Add a table at the bottom:

| Symptom | Likely Cause | Fix |
|---|---|---|
| 401 on every request | Secret/token mismatch | Re-set `WEBHOOK_SECRET` or `ARGOCD_WEBHOOK_TOKEN` via `forge variables set` |
| No deployments, no errors | Missing `jira` annotation | Add `jira` annotation to HelmRelease/Application |
| 400 "Missing env annotation" | Missing `env` annotation | Add `env` annotation |
| Environment shows as "unmapped" | Missing `env-type`/`envType` | Add the annotation with a valid Jira environment type |
| Deployment panel link broken | Missing or empty `url` | Add `url` annotation |
| 502 errors | Jira API issue | Check `forge logs`, verify app scopes |
